# dr-exam-log

A cross-platform CLI tool that monitors the active window (foreground application or browser tab) and logs state transitions to a timestamped file. Designed for exam monitoring scenarios where a proctor needs to verify that a student stays within permitted applications and websites.

---

## Purpose

The project solves the problem of recording which application or browser tab a user has focused on over time. It writes a chronological log of every **transition** (switch) between windows, apps, or web pages. This log can be reviewed after an exam session to detect unauthorized activity.

It is intended for:
- **Faculty / exam proctors** who need an auditable record of student screen activity during online exams.
- **Students** who run the tool themselves to prove they stayed within exam rules.
- **Developers** who need a lightweight, cross-platform foreground-window observer.

---

## Features

- **Cross-platform active window detection** via the [`active-win`](https://github.com/sindresorhus/active-win) library — works on Windows, macOS, and Linux.
- **Windows‑augmented detection** — on Windows, a PowerShell script using `UIAutomationClient` and Win32 API (`GetForegroundWindow`, `GetWindowText`) provides a secondary reading of the foreground window, which is merged with the `active-win` result for higher reliability.
- **Browser URL capture** (Windows only):
  - Reads the URL from the browser's address bar via Windows UI Automation.
  - Falls back to Chrome DevTools Protocol (`http://127.0.0.1:9222/json/list`, ports 9222/9223/9229) when Chrome is in the foreground.
- **Browser tab name inference** — strips known browser suffixes (` - Google Chrome`, ` - Mozilla Firefox`, etc.) from the window title to extract the tab name.
- **URL inference from window title** — recognizes known patterns (e.g., "ChatGPT", "YouTube", "Gmail", "Google", "Darshan/GNUMS", "DeepSeek") and maps them to canonical URLs.
- **App‑friendly labels** — common process names are mapped to human‑readable names (e.g., `code` → `VS Code`, `explorer` → `File Explorer`, `chrome` → `Chrome`).
- **State deduplication** — consecutive identical states are not logged; only actual transitions produce a log line.
- **Timestamped log format**: `YYYY-MM-DDTHH:mm:ss | current_state ----> next_state`
- **Configurable polling interval** (default 1 s, minimum 0.2 s).
- **Configurable output path** via CLI flag or `DR_EXAM_LOG_OUTPUT_DIR` environment variable.
- **Zero runtime dependencies** beyond `active-win`.
- **Available via `npx`** — no global install required.

## Architecture

### Overall design

The tool is a single‑process Node.js script with a synchronous polling loop. No server, no database, no background workers. It runs in the foreground until the user presses Ctrl+C.

### Detection flow on Windows

1. `collectWindowsFocusContext()` runs a PowerShell script that calls `GetForegroundWindow()` → `GetWindowText()` → `GetProcessById()` to obtain the foreground window title and process name.
2. Simultaneously (or in fallback), `activeWindow()` from the `active-win` library retrieves the same information.
3. The two results are **merged**: the PowerShell result is preferred for title and process name, but missing fields are filled from the `active-win` result.
4. If the detected process is a browser and no URL was found, `collectChromeUrlFromDevTools()` attempts to reach Chrome's DevTools Protocol endpoint (`http://127.0.0.1:9222/json/list`) to read the active tab URL.
5. `resolveState()` classifies the result into one of four state kinds:
   - `web` — browser is foreground and a URL is known.
   - `browser` — browser is foreground but no URL is known (logs tab title only).
   - `app` — a non‑browser application is foreground.
   - `window` — a window title is available but the process could not be named.
   - `unknown` — nothing could be determined.
6. A transition is written only when the state **key** changes (i.e., the deduplication identifier differs from the previous poll).

### Detection flow on macOS / Linux

1. `activeWindow()` from the `active-win` library provides the foreground window title, process name, and (when supported) browser URL.
2. No platform‑specific scripts are executed.
3. The same `resolveState()` logic classifies and deduplicates.

### State transition logging

The `StateTransitionLogger` class:
- Opens (or creates) a log file at the configured path.
- Writes a header line if the file is new or empty.
- On each `observe(state)` call, checks whether `state.key` differs from `lastKey`.
- On a change, appends: `{ISO timestamp} | {previous state value} ----> {new state value}`
- On no change, silently returns `false`.
- All writes use `fs.appendFileSync` — synchronous I/O is acceptable because the polling interval (≥200 ms) is much larger than the write time.

## How It Works

1. The user runs `npx dr-exam-log` (or the local equivalent).
2. The CLI parses optional arguments (`--interval-seconds`, `--output`, `--help`).
3. A `StateTransitionLogger` is created, which opens (or creates) the log file at the default path (`~/Downloads/proctor_state_transitions.log`) or the path specified by `--output` / `DR_EXAM_LOG_OUTPUT_DIR`.
4. The main polling loop begins:
   - **Every N seconds** (default 1, minimum 0.2), the tool determines the current foreground window.
   - On **Windows**, this is a two‑step process: first a PowerShell script reads the foreground window directly via Win32 API (`GetForegroundWindow`, `GetWindowText`), and then `active-win` is called as a fallback. The results are merged. If Chrome is detected and no URL was found, Chrome DevTools Protocol is queried.
   - On **macOS / Linux**, `active-win` is the sole source of foreground window information.
5. The raw window data (`title`, `owner`, `url`) is passed to `resolveState()`, which:
   - Normalizes the process name.
   - Detects whether the foreground window is a browser.
   - If a browser URL is available, classifies the state as `web` with a value like `ChatGPT(https://chatgpt.com)`.
   - If a browser is foreground but no URL is known, classifies as `browser` with the tab title.
   - Otherwise classifies as `app` with the friendly process name.
6. The state is passed to `logger.observe()`. If the state **key** (a canonical deduplication string) differs from the previous poll, a line is appended to the log file and printed to stdout.
7. The loop sleeps for the remainder of the interval (accounting for detection time) and repeats.
8. When the user presses **Ctrl+C**, the `SIGINT` handler sets a flag that causes the loop to exit cleanly.

### Why some features are Windows-only

- The PowerShell‑based UIAutomation and Win32 interop are inherently Windows-specific.
- Chrome DevTools Protocol detection is only attempted on Windows because the primary URL capture on macOS and Linux relies on what `active-win` provides out of the box.

### Known limitations

- **Linux / Wayland**: `active-win` has limited visibility on Wayland sessions. X11 is well supported.
- **macOS**: Browser URL capture depends on OS permissions and may not work in all browsers.
- **Chrome DevTools Protocol**: Requires Chrome to be launched with `--remote-debugging-port` or the `--remote-debugging-port` flag enabled at runtime. The tool silently falls back to tab‑title‑only logging when the endpoint is unreachable.
- **No encryption / signing**: The log file is plain text. A motivated user could modify it. This tool provides a log of record, not a cryptographic proof.
